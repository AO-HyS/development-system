# AOHYS Development System contract

Contract version: `0.4.0`

## Observable harness adapters

Codex and Factory are native adapters behind one Development Contract Scenario Interface. Each adapter maps the common lifecycle to its real tools, agent/droid roster, models, reasoning controls, hooks, discovery rules, external state, and side-effect boundary. Adapter parity is decided from observable stage selection, transition, authorization, external side effects, evidence, and terminal state; filenames and command names are not parity evidence.

T3Code is a Codex client surface, not a third semantic adapter. It uses the Codex executable contract and state namespace. Operational validation reports any observable divergence before T3Code can be considered independently adaptable.

The operational validator launches the installed Codex and Factory executables in read-only mode across AO root, a simple repository, a mature repository, and a nested CWD. Its scenarios inspect instructions, catalog exposure, actual load, hooks, roster, model/role, side effects, external state, and overflow. Failures are attributed to canonical source, adapter translation, harness runtime, or repository policy.

## Capability benchmarks and roster

The permanent benchmark suite compares identical fixture text, instructions, and checks for orchestration, implementation, review, architecture, browser QA, and visual judgment. Every run records the fixture hash, harness, model, reasoning, duration, correction time, verified-delivery time, tokens, reported cost, corrections, findings, and slop.

Rankings are per capability and never declare a universal winning model. GPT-5.6 Sol, Spark, and Factory alternatives remain baselines or challengers until repeatable results support a roster change. The roster maps capabilities to roles and harness-specific model/reasoning settings. Factory `inherit` is valid only with an explicit resolved model, so reruns never depend on an invisible parent session.

## Implement Preview delivery loop

An approved Implement Preview may execute only its exact terminal slice. One writer is the default. Intent and standards reviews run in independent clean processes; specialized jurors are configured only when the plan requires them. Findings use blocker, high, medium, and low severity. Completion requires no blocker/high and an explicit disposition for medium findings. Repeated blocking fingerprints pause the loop as non-convergent instead of converting failure into success.

TDD and browser/computer QA are proportional decisions with recorded reasons and evidence. An omission requires alternative evidence. The authorized loop may implement, test, validate, review, correct, perform QA, commit, push, open/update a pull request, and publish a preview. It cannot merge, release, or deploy production.

Local Visual Plan and Local Visual Recap are private files outside the target repository and pull request. The recap includes failures and corrections, residual risk, PR and preview links, and a manual checklist. The terminal state is `ready-for-human`; merge, release, and production still require separate operation-specific authorization.

## Prior guarantees retained

The `0.3.0` lifecycle state machine, persisted human gates, exact one-shot sensitive-operation grants, canonical installation, skill provenance, six-state skill evidence, drift detection, reversible synchronization, and rollback guarantees remain in force. Generated HOME state cannot redefine the canonical repository, and validation never writes to the operator's HOME unless an explicit install, sync, or authorized delivery operation requests it.
